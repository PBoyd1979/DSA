UML overview

UML - Unified Modelling Language

Based on a collaboration of Grady, Booch & Rumbaugh (The Three Amigos) and interest from the big three at the time Microsoft, Oracle and IBM in the mid 90’s. A book was published called ‘The Unified Modelling Language User Guide’ in 1999 providing the industry standard we use today.
The basis of UML came from a number of other notions:

  • Object Orientated Design
  • Object Modeling Technique
  • Object Orientated Software Engineering

The purpose of UML was to:

  • Provide software developers a similar process and language to communicate ideas
  • Offering visual tools and user focused techniques

Digital Skills Academy has produced a simple Infographic that shows the UML process which was used in the development of Medic!

Use cases

Use case 1: Traveler registers on Medic!

Primary actor: Traveler (Medic User)

Brief: The traveler downloads the Medic! app onto their device and registers to become a user. They provide their contact details and begin to build up a profile on the system.

Pre-conditions: The traveler has a device that is capable of connecting to the internet and has capacity to download and install an application. The user has set up an account on the Medic! web site or through an app portal.

Post-conditions: The traveler has downloaded and installed Medic! onto their device. They have registered as a user and have entered some background information to their profile.

Triggers: The traveler decides they need easy access to medical advice or assistance while away from home, they decide to install the Medic! app in order to get access to this information.

Normal course:

  1. The Traveler visits the App store and downloads the Medic! app onto their device.
  2. The systems requires that the User has registered.
  3. A portal is available to the User to create an account.
  4. The to access the Medic App the user must log in.
  5. The user enter into the home screen.
  6. The user engages with the App answering a series of questions.
  7. The App continues asking questions until it has sufficient information to provide a diagnosis.
  8. The App also can use information from the users profile and third party apps and wearables.
  9. The user can then save the diagnosis and get it translated into another language of their choosing.
  10. The user also has the option to to look for medical assistance in their vicinity.

Use case 2: Traveler gets diagnosed

Primary actor: Traveler

Brief: The traveler becomes unwell while away from home, they input their symptoms into Medic! and receive a diagnosis and recommended course of action.

Pre-conditions: The traveler has Medic! installed on their device and has connection to the internet and has a device that has GPS.

Post-conditions: The traveler has input their symptoms and received a diagnosis and recommended course of action.

Triggers: The traveler becomes unwell while in a foreign country and they decide to seek assistance using the Medic! app that they have access to on their device.

Normal course:

  1. The User has received a diagnosis.
  2. The User can ask the app to show him/her information on where to find medical help.
  3. The User then can select to email, call or travel to the healthcare provider
  4. The user is taken to a map page showing all local healthcare providers
  5. All health care providers are rated by users and offer information on the them.
  6. The user can read reviews of Health providers posted by other users.
  7. When the User selects email the app utilised the mobile devices mail app and attached a repot in the language of choice.
  8. When the User selects call the app utilises the mobile devices phone to present a number that can be dialled to contact the desired healthcare provider.
  9. When the User selects the navigation option it plants the best route to that location.
  10. The Medic! App will utilise the mobile devices GPS to assist.

Use case diagrams

Use case 1: Get Diagnosed

Purpose: How Medic! user gets diagnosed when using the app

Summary

The user does not feel well and logs into their account, the application on opening they are sent to main screen allowing them to be offered questions. The questions are produced by the Medic! logic tree algorithm, and continue to be asked until diagnosis is provided, all one click with no typing. The Medic! app also uses additional info from 3rd party devices such as wearables etc. Info is also provided by user in their profile when they set up their account. When Medic! has exhausted all of its logic algorithms a list of recommended treatments is provided to the user. Including a report can be produced and translated into another language, saved to the user's account. The user can also can choose to go to a local map which provides medical assistance and relevant information to allow the user to go and get treatment or purchase medication.

Use case 2: Get fixed

Purpose: Use Medic! app to get treatment

Summary

The user reviews treatment recommendations and can review detailed diagnosis on device. A report can be produced and translated into another language, saved to the users account. The user can also can choose to go to a local map (email, address, phone number, Medic! rating) which provides medical assistance and relevant information to allow the user to go and get treatment or purchase medication. The user selects which option they what i.e option to email (produce report, option to get translation), option to call, option to navigate to a service provider.
